# Web Crawler Implementation Standards

This document establishes the universal standards for developing and maintaining web crawlers within this repository. All new crawlers and major revisions must align with these guidelines to ensure performance, stability, and anti-bot resilience.

## 1. Core Principles

Crawlers are responsible for parsing third-party e-commerce retail data reliably. The primary focus should be on **resiliency against layout changes** and **mimicking legitimate browser traffic**. 

The repository supports two distinct crawling paradigms:
1. **HTTP Client (Default)**: Fast, lightweight, and uses static HTML parsing.
2. **Headless Browser (Fallback)**: Used only when JavaScript execution or advanced anti-bot bypass (e.g., Cloudflare) is strictly required.

## 2. HTTP Client Standards

To maintain parity and prevent fingerprinting discrepancies across the codebase, strictly adhere to the project's standard HTTP configuration.

### The Standard Client: `curl-cffi` + `browserforge`

- **Preferred Client**: All crawlers **MUST** utilize `curl_cffi.requests.AsyncSession` for executing web requests. Do not introduce alternatives like `httpx` or `aiohttp` for external parsing tasks.
- **Fingerprint Spoofing**: Use `browserforge.headers.HeaderGenerator` injected into the `AsyncSession` to bypass basic bot mitigation (TLS and Header fingerprinting).

> [!IMPORTANT]
> Refer to the `create_client()` utility in `app/crawlers/http_client.py` for the central implementation of this stack. 

### Middleware and Network Resilience

- **Retry Protocols**: Crawlers must implement resilient fetching logic. When utilizing the HTTP client, you should standardise retries on ephemeral network errors (e.g., Timeout, 502, 503, 504 errors) using an explicit retry decorator or backoff generator.
- **Timeouts**: Define explicit request timeouts for all HTTP calls. Do not allow requests to hang indefinitely.
- **Header Rotation**: Always generate a new User-Agent/Header signature per session (or per significant batch) rather than using a hardcoded static User-Agent.

### Strict Typing Expectations

Data parsing should strictly map to static schemas to easily catch changes in structure.

- Enforce the use of **Pydantic** models for all return values produced by a crawler (e.g., creating a standardized `ProductSearchResult` model).
- All output models **MUST** include a `source` field indicating the retailer. This field should be defined as a `Literal` with a default value matching the provider name.
- Crawler functions **MUST** return instantiated Pydantic models, not raw dictionaries.
- Define formal type hints for all Request/Response cycles and payload parsing methods.

## 3. Selector Best Practices

Extracting data from an HTML document must prioritize the most stable anchor available. Crawlers should utilize `parsel.Selector` or `lxml` explicitly. 

### Primary Paradigm: Semantic Targeting
Prioritize data stability above all else. Target standard attributes intentionally placed by the development teams of the scraped site.
- **Attributes**: Look for `data-product-id`, `data-test-id`, `data-sku`, or specific `aria-labels` attached to elements.
- **Rationale**: These data attributes rarely change during stylistic CSS sweeps/builds.

### Secondary Paradigm: Metadata Over Styling
If explicit semantic attributes are missing, target the invisible structural data of the document before reading the UI itself.
- **Open Graph / Meta Tags**: Prefer extracting core titles, descriptions, and hero images from `<meta property="og:title" ...>` or `<meta name="twitter:image" ...>` instead of targeting `<h1>` or `.hero-img` in the body.
- **JSON-LD/Schema.org**: Exploit embedded `<script type="application/ld+json">` data tags. Many hardware stores embed complete product schemas directly in the page which avoids all visual parsing entirely.

> [!WARNING]
> **Avoid Brittle CSS Paths**: Explicitly avoid relying on deep UI paths (e.g., `div.container > main > section.products > ul:nth-child(2) > li.item-hash-xyz > span`). 
> **Avoid Obfuscation**: Do not target obfuscated utility classes generated by frameworks like Tailwind or Styled-Components (e.g., `.flex.mt-2.text-sm` or `.css-1h2p3z`).
> **Requirement**: If you absolutely *must* rely on brittle CSS selectors due to a lack of better alternatives, you **MUST** explicitly document this choice in your final `walkthrough.md` during implementation. 

### Tertiary Paradigm: Defensive Parsing
Provide robust fallbacks for dynamic implementations.

- Always provide `.get(default=None)` or equivalent safe access patterns. Do not assume any tag or structure will always be present or list out of bounds without attempting a rescue.
- Normalize whitespace (`\n`, `\t`, `\r` and multiple spaces) and cleanly strip trailing elements using utility modules (like those in `app/crawlers/utils.py`).

## 5. Browser-based Crawling

When a site's data is heavily dependent on JavaScript or protected by sophisticated anti-bot challenges that `curl-cffi` cannot bypass, use the browser-based stack.

### The Browser Stack: `playwright` + `browserless`

- **Client**: Use `playwright` connected to a `browserless` service for CDP-based interaction.
- **Authentication**: The stack requires a `BROWSERLESS_API_KEY`. 
    - Locally, `scripts/test.sh` fetches this automatically from `gcloud` secrets.
    - If running manually, use: `export BROWSERLESS_API_KEY=$(gcloud secrets versions access latest --secret="browserless-token")`.
- **Context Management**: Always use the `create_browser()` utility from `app/crawlers/browser.py` to ensure consistent session handling and connection timeouts.
- **Locators vs. Selectors**: When in browser mode, prioritize using native Playwright **Locators** (`page.locator()`) instead of `parsel.Selector`. Locators are more robust as they can wait for elements to be visible/attached automatically.

### Strategic Implementation
- **Wait Strategies**: Avoid hardcoded sleeps. Use `wait_for_selector()` or `wait_for_load_state("networkidle")` with explicit timeouts.
- **Timeouts**: Define generous timeouts (e.g., 60-90 seconds) for initial page loads to account for solver delays.

## 6. Testing Requirements

A crawler is only complete when it adheres to the **Spec Driven Development (SDD) Testing Protocol**.

### HTTP Crawler Testing (Mocked)
- **Standard**: All HTML fixtures must be captured manually and served locally using `pytest-httpserver`. 
- **Rule**: **No live API requests** should be made during regular test suite execution for HTTP crawlers.
- **Reference**: Review [Testing Rules](file:///Users/david/apps/uk-hardware-store-mcp/.agents/rules/testing.md).

### Browser Crawler Testing (Live/Skipped)
- **Standard**: Browser-based crawlers often require live connections to test interaction with anti-bot solvers.
- **Requirements**:
    - Tests **MUST** be generated for every browser crawler to verify structural parsing.
    - Tests **MUST** be decorated with `@skip_if_ci` (from `tests/__init__.py`) to prevent execution in CI environments where secrets/browserless might be unavailable.
    - These tests are intended for local development and manual verification only.
